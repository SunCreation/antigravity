Why Prompting

왜 프롬프팅이 중요한가?

바이브 코딩에서 프롬프트 = 코드입니다.

입력의 품질이 출력의 품질을 결정합니다 — Garbage In, Garbage Out.

  • 같은 AI 모델이라도 프롬프트에 따라 결과가 크게 다름
  • 구체적인 프롬프트 → 정확한 코드
  • 모호한 프롬프트 → 엉뚱한 코드

Anatomy

좋은 프롬프트의 3요소

📍
Context
맥락
현재 상황, 프로젝트 정보, 관련 파일
🎯
Intent
의도
무엇을 하고 싶은지, 최종 목표
📏
Constraints
제약조건
기술 스택, 스타일, 금지사항
✅
공식
좋은 프롬프트 = Context + Intent + Constraints

Anti-patterns

프롬프트 안티패턴 4가지

  • 너무 짧음 — "버튼 만들어줘" → 어떤 버튼? 어디에?
  • 맥락 없음 — "에러 고쳐줘" → 어떤 에러? 어디서?
  • 모순된 지시 — "심플하게, 근데 화려하게"
  • 한번에 너무 많이 — "전체 앱 다 만들어줘"

Refinement

모호한 프롬프트 → 구체적 프롬프트

모호한 프롬프트 ❌구체적 프롬프트 ✅
요청"로그인 만들어줘""React + TS로 이메일/비밀번호 로그인 폼. 유효성 검사 포함."
디버그"에러 나""useAuth 훅에서 401 에러. 토큰 만료 처리 로직 추가해줘."
수정"이거 좀 이쁘게""Tailwind로 카드에 그림자와 둥근 모서리 추가해줘."

Principle

Think-First: 생각 먼저, 코드는 나중에

"코드를 짜기 전에 계획을 세우세요. AI가 계획을 세우게 하세요."

AI에게 즉시 코드를 작성하라고 하지 말고, 먼저 설계와 분해를 요청하는 워크플로우

Step 1

Step 1: 요구사항 명확히 설명하기

AI에게 "무엇을 만들 것인지"를 구체적으로 전달합니다.

💬
프롬프트 예시
"사용자 대시보드를 만들고 싶어.
• 사용자: 관리자와 일반 사용자
• 주요 기능: 실시간 통계, 사용자 목록, 알림
• 기술: React + Tailwind"

Step 2

Step 2: 계획 수립 요청하기

AI에게 코드 대신 계획을 먼저 달라고 요청합니다.

💬
핵심 프롬프트
"이 요구사항을 구현하기 위한 단계별 계획을 세워줘. 아직 코드는 작성하지 마."
💡
왜 효과적인가?
AI가 문제를 분해하는 과정을 볼 수 있고, 방향이 맞는지 먼저 검증할 수 있습니다.

Step 3

Step 3: 계획 검토하기

  • 빠진 단계가 없는지 확인
  • 각 단계가 독립적인지 확인
  • 순서가 논리적인지 검증
  • 우선순위가 적절한지 조정
⚠️
주의
이 단계를 건너뛰면, 나중에 전체를 다시 해야 할 수도 있습니다. 여기서 5분이 나중에 50분을 아껴줍니다.

Step 4: 점진적으로 실행하기

Decomposition

분해: 큰 문제 → 작은 단위

복잡한 기능은 반드시 서브태스크로 나눕니다.

🔪
분해 원칙
각 태스크는 독립적으로 테스트 가능해야 함
📏
적절한 크기
하나의 태스크 = 하나의 PR 정도의 크기
🔗
의존성 파악
어떤 순서로 구현해야 하는지 파악

Example

예시: "대시보드 만들어줘" → 8개 서브태스크

  1. 프로젝트 구조 및 라우팅 설정
  2. API 데이터 페칭 레이어 구현
  3. 통계 카드 컴포넌트 개발
  4. 사용자 목록 테이블 구현
  5. 알림 시스템 구축
  6. 반응형 레이아웃 적용
  7. 로딩/에러 상태 처리
  8. 접근성(a11y) 검토

Best Practices

이미지 프롬프팅 베스트 프랙티스

✅
좋은 예시
"이 Figma 디자인을 Next.js + Tailwind로 구현해줘. 다크모드, 모바일 반응형 포함."
❌
나쁜 예시
[이미지만 첨부하고 아무 설명 없음]
  • 이미지는 고해상도로 (흐린 스크린샷 피하기)
  • 항상 텍스트 설명을 함께 제공
  • 기술 스택과 요구사항을 명시
  • 반응형 여부를 명확히 전달

Pattern 1

"Explain this code first, then..."

AI에게 먼저 설명하게 하고, 그 다음에 작업을 요청합니다.

💬
프롬프트
"이 파일의 구조를 먼저 설명해줘. 그 다음에 [원하는 수정사항]을 적용해줘."
변형 1
"이 아키텍처를 요약해줘"
변형 2
"이 함수의 사이드이펙트를 분석해줘"

Pattern 2

"Create a plan for X, don't code yet"

AI에게 실행 계획을 먼저 세우게 합니다.

💬
프롬프트
"사용자 인증 시스템을 구현할 단계별 계획을 세워줘. 아직 코드는 작성하지 마."
💡
마법의 문장
"don't code yet" — 이 한 줄이 AI를 구조적 사고 모드로 전환합니다.

Pattern 3

"Refactor Y following Z pattern"

리팩토링 시 패턴과 원칙을 명시합니다.

💬
프롬프트
"UserService를 Repository 패턴으로 리팩토링해줘. 기존 동작은 그대로 유지하고, 테스트도 같이 수정해줘."
대상
어떤 코드를?
방향
어떤 패턴으로?
제약
무엇을 유지?

Pattern 4

"Write tests for X"

테스트 작성 시 프레임워크와 범위를 명시합니다.

💬
프롬프트
"auth.ts의 login 함수 단위 테스트를 Vitest로 작성해줘. 정상 케이스 3개, 에러 케이스 5개, 커버리지 90% 이상."
🎯
좋은 테스트 프롬프트 요소
테스트 도구 명시 · 범위(단위/통합) · 케이스 유형 · 목표 커버리지

Pattern 5

"Debug: error + context"

디버깅 시 에러 + 컨텍스트를 함께 제공합니다.

💬
프롬프트
"TypeError: Cannot read property 'user' of undefined 발생. src/hooks/useAuth.ts 42번째 줄. 로그인 직후에만 에러. 원인 분석하고 수정해줘."
❌
나쁜 예시
"에러 나요 고쳐줘"
✅
좋은 예시
에러 메시지 + 파일 위치 + 발생 조건

Model Choice

언제 Flash를, 언제 Pro를?

⚡
Gemini Flash
• 빠른 반복 작업
• 간단한 코드 수정
• 파일 설명 요청
• 단순 버그 수정

→ "빠르게 확인하고 싶을 때"
🧠
Gemini Pro / Claude
• 아키텍처 설계
• 복잡한 리팩토링
• 장문 코드 생성
• 다중 파일 수정

→ "깊이 있는 분석이 필요할 때"
💡
전략
Flash로 빠르게 프로토타입 → Pro/Claude로 품질 개선
